# Especificación: regla Write-Ahead Log

> **Issue:** #41
> **Milestone:** 07 Write-Ahead Log
> **Estado:** tested.

## Propósito

Este paso documenta la regla central de WAL: una página no debe modificarse
antes de que exista en el log el registro que explica esa modificación.

## Alcance actual

- `docs/07-write-ahead-log.md` explica el orden correcto: construir `Update`,
  agregarlo a `WriteAheadLog` y después aplicarlo sobre `PageStore`.
- README y ROADMAP elevan el capítulo 07 a `tested`.
- El checklist del plan marca completo el capítulo Write-Ahead Log.
- No se agrega una API nueva porque `WriteAheadLog`, `LogOperation::Update` y
  `PageStore::redo` ya permiten mostrar el flujo educativo.

## Regla

La secuencia canónica del modelo es:

1. Leer o conocer la imagen `before`.
2. Construir una operación `Update` con `before` y `after`.
3. Agregar la operación a `WriteAheadLog` para obtener un LSN.
4. Aplicar el `after` sobre `PageStore` mediante redo educativo.

El orden inverso queda fuera del contrato conceptual del capítulo: si la página
cambia antes de que exista su registro WAL, recovery no tiene una historia
confiable para explicar ese estado.

## Decisión de diseño

La regla se documenta como invariante conceptual, no como garantía de
durabilidad física. El curso todavía no modela `fsync`, disco real, buffer pool,
checkpoints ni recuperación después de crash. Esos elementos pertenecen al
capítulo de Recovery y a fases posteriores del camino.
